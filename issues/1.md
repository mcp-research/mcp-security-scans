### Title: Fix pip --require-hashes Failure Due to Unpinned Dependency (exceptiongroup) in requirements-dev.lock

---

**Describe the bug**\nThe workflow run (see below) fails during dependency installation because pip, in --require-hashes mode, detected a dependency (exceptiongroup) specified without an exact version pin. The lock file includes exceptiongroup as ≥1.0.2 (transitive from anyio), whereas pip requires all dependencies to be pinned (with ==) when hashes are used.

**Workflow run failure:**  \nhttps://github.com/mcp-research/mcp-security-scans/actions/runs/22805536503/job/66154232306?pr=184

**Error excerpt:**  \n```
ERROR: In --require-hashes mode, all requirements must have their versions pinned with ==. These do not:
    exceptiongroup>=1.0.2 ... (from anyio==4.12.1->-r requirements-dev.lock)
```

---

**Steps to Reproduce**\n1. Run the affected workflow on the current branch.\n2. Observe pip fail on dependency installation due to improper pinning/hashing.

---

**Proposed fix**\nRegenerate the requirements-dev.lock file to ensure all dependencies (including transitive ones such as exceptiongroup) have exact version pins (==) and hashes.  \nYou can do this with:\n```
pip-compile --generate-hashes --output-file=requirements-dev.lock requirements-dev.txt
```
This will refresh the lock file and resolve the installation error.

---

**Additional notes**\nIf transitive dependencies are not being pinned, review requirements-dev.txt and consider pinning anyio or exceptiongroup directly if necessary for reproducibility.
